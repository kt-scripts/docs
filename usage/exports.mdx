Exports allow other resources on your server to call KT Extra Menu **programmatically** — without the player having to type a chat command or press a key. This is particularly useful when you want to open the menu from an interaction script (e.g. `ox_target`, `qb-target`) or need custom trigger logic.

## Available exports

KT Extra Menu registers three client-side exports:

| Export | Description |
| --- | --- |
| `OpenVehicleMenu` | Opens the full vehicle customization menu (main menu with livery and extras selection) |
| `OpenLiveryMenu` | Opens the livery submenu of the current vehicle directly |
| `OpenExtrasMenu` | Opens the extras submenu of the current vehicle directly |

---

### OpenVehicleMenu

Opens the **main menu** of KT Extra Menu. The player first sees the choice between liveries and extras.

```lua
exports['kt_extra_menu']:OpenVehicleMenu()
```

---

### OpenLiveryMenu

Opens the **livery submenu** directly, without the player having to go through the main menu first. The script automatically determines the player's current vehicle.

```lua
exports['kt_extra_menu']:OpenLiveryMenu()
```

---

### OpenExtrasMenu

Opens the **extras submenu** directly. The player's current vehicle is also determined automatically here.

```lua
exports['kt_extra_menu']:OpenExtrasMenu()
```

---

### All three exports at a glance

```lua
-- Open main menu (liveries + extras selection)
exports['kt_extra_menu']:OpenVehicleMenu()

-- Open only the livery menu
exports['kt_extra_menu']:OpenLiveryMenu()

-- Open only the extras menu
exports['kt_extra_menu']:OpenExtrasMenu()
```

<Note>
  All three exports enforce the **same restrictions** as the regular chat command: The player must be inside a vehicle, must be the driver when `Config.OnlyDriver = true`, and their job must be in `Config.RequiredJob` (unless the list is empty). Using an export does not bypass any of these checks.
</Note>

<Note>
  `OpenLiveryMenu` and `OpenExtrasMenu` determine the player's current vehicle **automatically**. You do not need to pass a vehicle handle. If the player is not in a vehicle, the export does nothing.
</Note>

## NetEvent as an alternative

In addition to the exports, there is also a registered **NetEvent** that triggers the main menu. This is convenient for resources that prefer events over exports — e.g. `qb-radialmenu`:

```lua
-- Open main menu via event (client side)
TriggerEvent('vehicle_customizer:openMenu')
```

This event is equivalent to `exports['kt_extra_menu']:OpenVehicleMenu()` and is subject to the same restrictions.

## Real-world example: ox\_target / qb-target

A typical use case is opening the menu through a vehicle interaction. The following example shows how to register a zone on a vehicle model using `ox_target`:

```lua
-- Example with ox_target (client side, e.g. in your interaction resource)
exports['ox_target']:addModel('police', {
    {
        name    = 'open_vehicle_menu',
        label   = 'Customize vehicle',
        icon    = 'fa-solid fa-spray-can',
        onSelect = function()
            exports['kt_extra_menu']:OpenVehicleMenu()
        end,
        distance = 2.0,
    }
})
```

```lua
-- Example with qb-target (client side)
exports['qb-target']:AddTargetModel('police', {
    options = {
        {
            type   = 'client',
            label  = 'Customize vehicle',
            icon   = 'fas fa-spray-can',
            action = function()
                exports['kt_extra_menu']:OpenVehicleMenu()
            end,
        }
    },
    distance = 2.0,
})
```

<Tip>
  Want to offer the livery menu and extras menu as **separate interactions**? Just use `OpenLiveryMenu` and `OpenExtrasMenu` as separate entries in your target configuration — that way, a single click takes you straight to the desired submenu.
</Tip>
